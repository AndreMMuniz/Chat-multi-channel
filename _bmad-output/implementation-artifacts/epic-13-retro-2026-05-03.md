## 1. Epic Review

### Epic Goal

Connect `Messages` and `Projects` through message-level contextual actions so operators can turn conversation content into structured work without leaving the inbox.

### Stories Reviewed

- `13.1` Message bubble context actions shell
- `13.2` Create card from message with project routing
- `13.3` Message tag, delete, and quick reply actions

### What Went Well

- The team landed the full bridge, not just the UI shell. The result is an end-to-end operational workflow:
  `Messages -> create card -> open in Projects -> return to source conversation`.
- The contextual `+` interaction stayed lightweight and visually integrated with the current inbox.
- Provenance was treated seriously. Cards created from messages preserve source message and conversation context.
- The team avoided fake frontend behavior and pushed the routing model into the backend with `project_context_id`.
- The workflow became easier to understand after adding visible message state:
  `Already a card`, direct `Open PRJ-...`, and contextual menu switching from `Create Card` to `Open Linked Card`.

### What Was Hard

- The original `Projects` domain was card-shaped, but Epic 13 forced a stronger distinction between:
  - conversation-linked project context
  - card creation inside that context
- This caused mid-epic structural work in the backend rather than simple UI wiring.
- There were several frontend build/type regressions caused by hook ordering and callback placement in `frontend/src/app/page.tsx`.
- Conversation tag updates exposed an enum mismatch between frontend and backend conventions:
  uppercase in the UI, lowercase in backend validation.
- SQLite test infrastructure became fragile after introducing cyclic FKs between `conversations`, `messages`, and `projects`.

### What We Learned

1. Message-to-project features cannot be implemented safely as UI sugar. They need explicit backend provenance and routing semantics.
2. Enum casing drift between frontend and backend is a real operational risk and should be normalized at the API boundary, not patched ad hoc in screens.
3. When a feature creates a bridge between two modules, success feedback and return-path navigation are part of the feature, not polish.
4. SQLite test fixtures need extra care once cyclic relationships are introduced, even if production runs on Postgres.

### Technical Debt / Residual Risk

- `Projects` still models the pipeline card itself as `Project`. If the product evolves to `project parent + task/card children`, this domain will need another architectural pass.
- The `Messages` page still has two known `<img>` warnings.
- There is still no E2E automation for the critical journey:
  `Messages -> create card -> Projects -> open source conversation`.
- The test fixture workaround for cyclic FKs is stable, but it is a symptom that the domain is getting more relationally complex.

## 4. Next Epic Preparation

### Suggested Next Epic Directions

Most natural options from this point:

1. `Tasks`
   Because the product is now able to convert conversation demands into tracked work, and the user previously described task-oriented use cases.

2. `QA / E2E Hardening`
   Because Epic 13 introduced a high-value cross-module workflow that deserves browser-level automated coverage.

3. `Projects Domain Evolution`
   Only if the team wants to evolve from `Project-as-card` into `Project context + child task cards`.

### Recommendation

The strongest near-term path is:

- short stabilization pass for QA/E2E around Epic 13
- then open the next product epic for `Tasks`

This preserves momentum without leaving the new bridge under-tested.

## 6. Final Summary

Epic 13 succeeded because it delivered a real operational bridge, not just contextual buttons. The team had to absorb unexpected backend/domain work, but the result is much more coherent: messages can become tracked work with preserved provenance, visible state, and navigable continuity across modules.

The main lesson is that cross-workspace workflows need backend truth, UX feedback, and return-path navigation together. Treating any one of those as optional creates confusion and rework.
